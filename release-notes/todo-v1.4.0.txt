      WHERE IT LIVES
      ──────────────

      Tone Trainer, Sentence Builder, Progress) — adding a 6th would make it
      cramped, especially on small screens. Instead, this becomes Level 6 in
      the existing 5-level mastery system, unlocked after the child earns 3★

      Rationale for Level 6 placement:
        - Tone Trainer = single-word tone recognition (simplest audio task)

        - AUDIO_TO_TRANSLATION quiz = single-word listen → pick meaning

          This is the hardest audio task and logically sits above all of
          the above, making it a natural mastery ceiling.

      HOW IT WORKS
      ────────────

      - A "Replay 🔊" button lets the child hear the sentence again (max 3
        replays per question to keep it a listening test, not a memory test).
      - 5 questions per run: drawn from the scenario's dialogue lines, not
        just the existing quizQuestions — so it uses all the content already
        in the JSON at no extra authoring cost.

[ ] 7b. Level 7 Mastery — Speaking Practice

      FEASIBILITY VERDICT
      ───────────────────
      Possible — but NOT via GBoard. Here is why, and what to do instead.

      WHY GBOARD IS THE WRONG TOOL
      ─────────────────────────────
      GBoard is a separate system app (an IME — Input Method Editor).
      Android provides no public API to:
        - Hide the keyboard layout while keeping only the microphone visible.
        - Force GBoard into voice-only mode programmatically.
        - Intercept or suppress its typing surface from another app.
      Any approach that tries to overlay or manipulate GBoard's window
      would be fragile, break on non-GBoard devices (Samsung, OPPO, etc.),
      and is likely to stop working on future Android versions.

      THE RIGHT TOOL: Android SpeechRecognizer API
      ─────────────────────────────────────────────
      Android ships a built-in SpeechRecognizer (android.speech.SpeechRecognizer)
      that is completely independent of the keyboard. It:
        - Records from the microphone directly.
        - Sends audio to Google's speech-to-text engine (same engine GBoard uses).
        - Returns simplified Chinese characters when locale is set to zh-CN.
        - Gives full programmatic control — no keyboard involved at all.
        - Works on ALL Android devices regardless of which keyboard is installed.
      This is how Duolingo, Google Translate, and every serious language app
      handle in-app speaking. It is the standard approach.

      REQUIRES INTERNET (first use) / OFFLINE MODEL (optional)
      ──────────────────────────────────────────────────────────
      The app already has android.permission.INTERNET declared.
      Speech recognition sends audio to Google's servers by default.
      Google also ships an offline Chinese speech model downloadable via
      Settings → Language & Input → Offline Speech Recognition.
      The app can detect offline availability and show a gentle notice
      if only online recognition is available (similar to the TTS voice
      pack notice in item 10).

      NEW PERMISSION REQUIRED
      ───────────────────────
      AndroidManifest.xml needs:
        <uses-permission android:name="android.permission.RECORD_AUDIO" />
      The app must request this at runtime (Android 6+). A one-time
      permission prompt is shown the first time the child reaches L7.

      WHERE IT LIVES
      ──────────────
      Level 7 in the mastery system — unlocked after 3★ on Level 6.
      The full mastery ladder becomes:
        L1  Standard quiz
        L2  12 targeted vocab questions
        L3  All words in scenario tested
        L4  Sentence Builder (multiple choice)
        L5  Sentence Builder (all sentences, no hints)
        L6  Listening Comprehension (hear sentence → pick meaning)
        L7  Speaking Practice (hear sentence → speak it back)

      HOW IT WORKS — FLOW
      ────────────────────
      1. A dialogue line from the scenario is selected (CHARACTER lines only).
      2. TTS speaks the line at normal speed so the child hears a model.
      3. A large animated 🎤 microphone button is shown.
         Tapping it starts recording (SpeechRecognizer.startListening).
      4. A waveform / pulsing ring animation plays while recording.
         Recording stops automatically when silence is detected
         (SpeechRecognizer handles this) or after a 6-second max.
      5. Recognized text (simplified Chinese characters) is compared to
         the expected textChinese.
      6. Score and feedback are shown (see Scoring below).
      7. 5 questions per L7 run, drawn from the scenario's CHARACTER lines.

      SCREEN LAYOUT (no TextField, no keyboard ever appears)
      ───────────────────────────────────────────────────────
      ┌────────────────────────────────────────┐
      │  [Character emoji + name]              │
      │                                        │
      │  Listen first:                         │
      │  ┌──────────────────────────────────┐  │
      │  │  🔊  "你今天吃了什么？"           │  │  ← tap to replay TTS
      │  │      Nǐ jīntiān chī le shénme?   │  │
      │  └──────────────────────────────────┘  │
      │                                        │
      │  Now you say it:                       │
      │                                        │
      │         [ 🎤  TAP TO SPEAK ]           │  ← large 56dp+ button
      │                                        │
      │  [waveform animation while recording]  │
      │                                        │
      │  [result / feedback card after done]   │
      └────────────────────────────────────────┘

      No text input field is ever shown. No keyboard is ever triggered.
      The mic button is the only interactive element during recording.

      SCORING — CHARACTER MATCH RATE
      ────────────────────────────────
      Compare recognised Chinese characters against expected textChinese
      using a character-level similarity score:

        score = (matching characters / total expected characters) × 100

      Thresholds:
        ≥ 90%  → ✅ Perfect / 很好！   — full marks
        70–89% → 🟡 Close / 再试一次   — half marks, encouraged to retry
        < 70%  → ❌ Try again / 加油！  — no marks, one free retry

      Why character match (not exact string match):
        - SpeechRecognizer may return traditional characters on some engines.
        - Particles (吗, 了, 的) are often dropped in natural speech.
        - An overly strict match would discourage children unfairly.

      ALTERNATIVE SCORING — PINYIN PHONETIC COMPARISON
      ──────────────────────────────────────────────────
      If character match proves too strict (e.g. engine returns wrong
      characters for homophones), fall back to pinyin comparison:
        - Convert expected textChinese → pinyin using the pinyinWords
          field already in the JSON (no extra library needed).
        - Convert recognized Chinese → pinyin using the same mapping.
        - Compare syllable sequences.
      This tolerates character-level errors while still checking pronunciation.
      Can be implemented as a second-pass fallback if the character score
      is below threshold.

      RETRY LOGIC
      ────────────
      - 70–89%: "Almost! Try again 🎤" button offered once.
      - < 70%:  Same retry offer, plus a "Skip this one" option so the
                child is never stuck.
      - After retry, best score of the two attempts is kept.

[ ] 13. Audio / TTS Not Working on Samsung One UI (reported: Samsung A51)
      DEVICE: Samsung A51 — Android 13, One UI 5.1, Kernel 4.14.113, Knox 3.9

      SYMPTOMS
      ────────
      - TTS audio does not play or plays silently during roleplay, flashcard,
        and tone trainer screens.
      - Likely caused by One UI's battery optimisation aggressively suppressing
        background audio, or the Samsung TTS engine being initialised instead of
        Google TTS, which may not have a Chinese voice pack installed.

      INVESTIGATION CHECKLIST
      ───────────────────────
      1. Detect active TTS engine at runtime (TextToSpeech.getDefaultEngine()) —
         log it in debug builds so we can see whether Samsung TTS or Google TTS
         is being used.
      2. Check TTS initialisation status in TtsManager — if status != SUCCESS,
         currently we silently fail. Show a one-time in-app notice guiding the
         user to install the Google Text-to-Speech engine + Chinese voice pack
         (ties into item 10 — Offline TTS Voice Pack Notice).
      3. Ensure TtsManager requests AudioFocus before speaking
         (AudioManager.requestAudioFocus) and releases it after — required on
         Samsung One UI 5+ to avoid being muted by the OS.
      4. Test against Samsung's Knox audio policy: Knox 3.9 can block certain
         audio streams in "secure mode". Add a fallback that plays audio on
         STREAM_MUSIC instead of STREAM_SPEECH if the latter is blocked.
      5. In AndroidManifest.xml confirm we are not declaring
         android:largeHeap="true" or aggressive doze-exempt flags that Samsung
         One UI penalises with audio throttling.

      DELIVERABLE
      ───────────
      - Robust TTS initialisation with visible error feedback (not silent failure).
      - AudioFocus acquisition/release in TtsManager.speak() and speakAndAwait().
      - One-time banner if Chinese TTS voice is absent (share logic with item 10).

[ ] 14. Bottom Navigation Bar Overflowing into System Nav Bar (reported: OPPO A60)
      DEVICE: OPPO A60 — Android 16, ColorOS, Kernel 5.15.180

      SYMPTOMS
      ────────
      - The app's bottom tab bar (5-tab nav in AppNavigation.kt) renders below
        the system navigation bar area (home / back / recents buttons), making
        the bottom tabs partially or fully untappable on gesture-nav devices.
      - Likely cause: window insets not applied — the app is not consuming
        WindowInsetsCompat.Type.navigationBars() so Compose draws into the
        system-reserved zone.

      INVESTIGATION CHECKLIST
      ───────────────────────
      1. In MainActivity.kt, ensure WindowCompat.setDecorFitsSystemWindows(window, false)
         is called so the app goes edge-to-edge and handles insets itself.
      2. The Scaffold or the BottomNavigation composable in AppNavigation.kt must
         apply Modifier.windowInsetsPadding(WindowInsets.navigationBars) or use
         Scaffold's contentWindowInsets parameter so content never overlaps the
         system nav bar.
      3. Verify that the NavigationBar composable uses
         NavigationBarDefaults.windowInsets (built into Material 3) — this is
         the most idiomatic fix and requires no manual padding.
      4. Test on both 3-button nav and gesture-nav modes on OPPO ColorOS, as
         the inset height differs significantly between the two.
      5. Also check StatusBar overlap at the top — same root cause can clip the
         top app bar on other OEM skins.

      DELIVERABLE
      ───────────
      - Edge-to-edge enabled in MainActivity with correct inset handling in
        Scaffold / BottomNavigation so no UI element overlaps system bars on any
        OEM skin or nav mode.
